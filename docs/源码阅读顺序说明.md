# 源码阅读顺序说明

这份说明是给第一次接触本项目源码的人准备的。

如果你一上来就直接看 `src/server/handle.c` 或 `src/client/client.c`，很容易被大量函数调用、协议细节和路径处理绕进去。更合适的方式是：先看“整个项目靠哪些基础模块搭起来”，再看“服务端怎么跑起来”，最后看“客户端怎么和服务端配合”。

下面给出一个比较适合初学者的阅读顺序。

---

## 一、先看项目整体结构

建议先浏览这些位置，不要求逐行看懂，先建立整体印象：

- `README.md`
- `项目概述.md`
- `项目描述文档.md`
- `docs/superpowers/specs/2026-04-17-phase1-design.md`

这一轮阅读的目标只有一个：先知道这个项目第一期到底要做什么。

你需要先搞清楚这几个问题：

1. 这个项目当前只做到第几期
2. 第一期支持哪些命令
3. 项目里同时有哪些模块：客户端、服务端、配置、日志、协议、线程池、会话、文件传输
4. 多用户隔离是怎么做的

如果这一步还没建立清楚，再往下读代码会比较吃力。

---

## 二、先看头文件，再看 `.c` 文件

对于 C 项目，一个很重要的阅读习惯是：

- 先看 `include/` 里的头文件
- 再看 `src/` 里的实现文件

原因很简单：

- 头文件更像“接口说明书”
- 源文件更像“具体施工过程”

如果不先看头文件，直接进 `.c` 文件，常常会看了半天还不知道一个结构体到底是干什么的、一个函数到底是给谁调用的。

建议先按下面顺序快速浏览头文件：

1. `include/common.h`
2. `include/config.h`
3. `include/log.h`
4. `include/protocol.h`
5. `include/queue.h`
6. `include/thread_pool.h`
7. `include/session.h`
8. `include/handle.h`
9. `include/file_transfer.h`
10. `include/client.h`
11. `include/socket_init.h`

这一步不用深究实现细节，只要先知道：

- 有哪些结构体
- 有哪些核心函数
- 每个模块负责什么

---

## 三、第一批重点：先看公共基础模块

真正开始看实现时，建议先看 `src/common/`。

阅读顺序建议如下：

### 1. `include/common.h` + `src/common/utils.c`

这是最基础的一层。

这里面主要是一些小工具函数：

- 去掉换行
- 去掉首尾空格
- 创建目录
- 检查名字是否合法
- 拆分命令

为什么先看它？

因为后面几乎所有模块都会用到这些函数。你先把这些工具函数的作用搞清楚，后面看到 `trim_space`、`split_command`、`ensure_dir` 时就不会反复卡住。

这一层重点理解：

- 为什么要先清洗输入字符串
- 为什么文件名和目录名要做合法性检查
- `split_command()` 是怎么把一整行命令拆成命令字和参数的

### 2. `include/config.h` + `src/common/config.c`

这个模块负责读配置文件。

建议边看代码边对照：

- `config/mycloud.conf`
- `config/users.conf`

你要重点理解：

- 主配置和用户配置为什么分成两个文件
- 为什么 `config_load()` 和 `config_load_users()` 分开实现
- 为什么项目里到处都是 `config_get_ip()`、`config_get_port()` 这种 getter，而不是直接操作全局变量

这一层看懂之后，你会知道：

- 服务端监听的 IP/端口从哪里来
- 线程池数量从哪里来
- 伪登录用户名从哪里来

### 3. `include/log.h` + `src/common/log.c`

这个模块主要是帮助你理解项目里的日志输出。

你要重点看：

- `log_init()`
- `log_write()`
- `LOG_INFO`、`LOG_ERROR`、`LOG_DEBUG`

这一层要弄明白：

- 为什么日志既打印到终端，又写入文件
- 为什么日志宏要自动带上文件名、函数名和行号
- 为什么日志级别低于当前配置时要直接跳过

### 4. `include/protocol.h` + `src/common/protocol.c`

这是最关键的公共模块之一。

如果把这个模块看懂了，后面的客户端和服务端通信就会顺很多。

这里要重点理解项目的协议格式：

- 文本消息：先发长度，再发内容
- 文件传输：先发文件大小，再发文件内容

重点函数：

- `protocol_send_n()`
- `protocol_recv_n()`
- `protocol_send_string()`
- `protocol_recv_string()`
- `protocol_send_file_size()`
- `protocol_recv_file_size()`

这一层一定要搞懂两个问题：

1. 为什么 `send` 和 `recv` 不能想当然地认为“一次就收发完整”
2. 为什么字符串要先发长度

这两个点弄懂了，你对整个项目通信流程的理解会提升很多。

---

## 四、第二批重点：看服务端的基础运行框架

公共模块看完后，建议开始看服务端，但不要一上来就看 `handle.c`。

更合适的顺序是：

### 1. `include/queue.h` + `src/server/queue.c`

先看队列模块，因为线程池是建立在队列之上的。

你要重点理解：

- 队列里放的到底是什么
- 为什么用链式队列
- 为什么主线程入队、工作线程出队

### 2. `include/thread_pool.h` + `src/server/thread_pool.c`

这一层是理解服务端并发模型的关键。

你要先看结构体 `thread_pool_t`，再看：

- `thread_pool_init()`
- `thread_pool_add_task()`
- `thread_pool_destroy()`

这里重点理解：

- 为什么线程池里要有互斥锁和条件变量
- 为什么主线程只负责“放任务”
- 为什么工作线程要自己循环等任务
- 为什么退出时要广播唤醒所有线程

### 3. `src/server/worker.c`

这个文件很短，但非常关键。

它相当于把线程池和真正业务逻辑接起来了。

你要重点理解：

- 为什么 `pthread_cond_wait` 外面要用 `while`
- 工作线程什么时候睡眠，什么时候被唤醒
- 为什么一个工作线程拿到 `client_fd` 后，要负责这个连接的整个会话

如果你把 `queue.c + thread_pool.c + worker.c` 连起来看懂，服务端的“并发框架”就基本清楚了。

### 4. `include/socket_init.h` + `src/server/socket_init.c`

这个模块比较单纯，就是网络初始化。

重点理解：

- `socket`
- `bind`
- `listen`
- `SO_REUSEADDR`

不用一开始就深究所有 socket 细节，但至少要知道：这个模块是专门把服务端监听套接字建起来的。

### 5. `src/server/main.c`

这个文件建议放在前面几个文件之后看。

因为当你已经知道：

- 如何创建监听套接字
- 如何初始化线程池
- 如何往线程池里放任务

再回来看 `main.c`，你会发现逻辑其实很清晰：

1. 读取配置
2. 初始化日志
3. 创建用户目录
4. 初始化线程池
5. 创建监听套接字
6. 循环 `accept`
7. 把 `client_fd` 投递给线程池
8. 退出时做清理

如果反过来先看 `main.c`，你会不断跳去找其他函数定义，阅读体验会差很多。

---

## 五、第三批重点：看服务端的业务层

这部分才是真正的“网盘功能”。

建议阅读顺序：

### 1. `include/session.h` + `src/server/session.c`

这个模块是理解“伪登录 + 虚拟路径”的核心。

你要重点理解结构体 `session_t` 中这几个字段：

- `client_fd`
- `is_login`
- `user_name`
- `current_path`
- `user_root`

再重点看这些函数：

- `session_init()`
- `session_login()`
- `session_build_current_real_path()`
- `session_build_target_path()`
- `session_cd_child()`
- `session_cd_parent()`

这一层要解决的是两个关键问题：

1. 用户当前处于什么状态，服务端保存在哪里
2. 客户端看到的虚拟路径，怎么映射成服务器上的真实路径

如果这一层没搞懂，后面 `handle.c` 会很难读。

### 2. `include/file_transfer.h` + `src/server/file_transfer.c`

这个模块建议在 `handle.c` 前面看。

原因是 `handle_puts()` 和 `handle_gets()` 最后都是调这个模块来真正收发文件。

你要重点理解：

- 上传时，为什么先收文件大小，再循环收文件内容
- 下载时，为什么先发文件大小，再循环发文件内容
- 为什么这里不直接在 `handle.c` 里写所有文件收发细节，而是拆成独立模块

### 3. `include/handle.h` + `src/server/handle.c`

这一层是服务端业务处理的核心，也是最值得慢慢看的一个文件。

建议不要一口气从头看到尾，而是按下面顺序看：

1. 先看 `handle_request()`
2. 再看 `handle_login()`
3. 再看 `handle_pwd()`、`handle_ls()`
4. 再看 `handle_cd()`、`handle_mkdir()`、`handle_rm()`
5. 最后看 `handle_puts()`、`handle_gets()`

这样读更自然，因为 `handle_request()` 就像一个总调度器。

它干的事情大概是：

1. 收客户端命令
2. 解析命令
3. 判断是否需要先登录
4. 分发到具体命令函数
5. 把结果回给客户端

看这个文件时，建议你边读边回答这些问题：

- 登录前为什么只允许 `login` 和 `quit`
- `pwd` 为什么最简单
- `ls` 为什么要过滤 `.` 和 `..`
- `cd` 为什么先检查目录存在，再修改 session
- `rm` 为什么只删普通文件
- `puts` 和 `gets` 为什么都要先发一条 `OK ready ...` 的确认消息

如果这些问题都能答出来，服务端业务层就基本读懂了。

## 六、第四批重点：看客户端

服务端看完后，再看客户端会更容易，因为你已经知道服务端想收到什么、会回什么。

建议顺序如下：

### 1. `include/client.h` + `src/client/command.c`

先看命令解析这一层。

这里相对简单，主要是：

- 命令切分
- 判断是否是第一期支持的命令

### 2. `src/client/file_transfer.c`

客户端上传下载和服务端文件传输模块是成对出现的。

建议你把这两个文件对照着看：

- `src/server/file_transfer.c`
- `src/client/file_transfer.c`

这样你能更容易看明白协议的两端是如何配合的。

### 3. `src/client/client.c`

这是客户端最核心的文件。

建议按下面顺序看：

1. `client_connect_server()`
2. `client_login()`
3. `client_read_command()`
4. `client_run_loop()`

重点理解：

- 客户端什么时候发命令
- 服务端什么时候回文本响应
- 上传时为什么客户端要等服务端先说“OK ready to receive file”
- 下载时为什么客户端收到 OK 后才开始收文件

### 4. `src/client/main.c`

这个文件最后看就行。

因为它本身就是个总入口，串起来的逻辑非常简单：

1. 读配置
2. 连接服务端
3. 登录
4. 进入命令循环

如果你前面都看明白了，这个文件几乎一下就能读懂。

## 八、阅读时建议边做的事情

只看代码不动手，理解会比较慢。建议你边读边做下面这些事：

### 1. 边读边画流程

例如自己在纸上画出：

- 客户端登录流程
- `puts` 上传流程
- `gets` 下载流程
- `cd` 命令处理流程

流程图不需要很正式，能帮助你理清“谁先发、谁后收”就够了。

### 2. 边读边跟日志

启动服务端后执行几条命令，再去看：

- `log/server.log`

这样你会更容易理解：

- 哪个命令到了服务端
- 服务端什么时候判定登录成功
- 服务端什么时候进入上传和下载逻辑

### 3. 边读边打断点或加打印

如果你在本地调试，最值得加打印的位置有：

- `handle_request()`
- `session_login()`
- `handle_puts()`
- `handle_gets()`
- `client_run_loop()`
- `protocol_send_string()`
- `protocol_recv_string()`

这样能帮助你把“代码逻辑”和“实际运行过程”对应起来。

## 九、初学者最容易卡住的几个点

### 1. 为什么字符串要先发长度

因为 TCP 是字节流，不会自动告诉你“一条消息到哪里结束”。

所以项目里用“先发长度，再发内容”的办法，明确告诉接收方接下来应该收多少字节。

### 2. 为什么线程池里要用互斥锁和条件变量

因为主线程和工作线程会同时访问队列：

- 主线程往里放任务
- 工作线程从里面取任务

没有锁就会乱。

而条件变量的作用是：队列为空时，让工作线程先睡眠，不要一直空转浪费 CPU。

### 3. 为什么要区分虚拟路径和真实路径

因为客户端不应该看到服务器真实目录结构。

客户端只应该看到自己的虚拟路径，例如 `/dir1`。

服务端内部再把它映射成真实路径，例如：

- `./data/users/alice/dir1`

### 4. 为什么一个工作线程要处理一个客户端连接的完整会话

因为这是第一阶段最简单、最稳定的实现方式。

这样做之后，这个线程手里一直拿着这个会话的上下文，不用把用户状态到处传来传去。

## 十、如果你已经读完第一遍，第二遍应该怎么读

第一遍阅读的目标是“知道代码在干什么”。

第二遍阅读建议带着下面这些问题去看：

1. 如果我要加一个新命令，应该改哪些文件
2. 如果我要把伪登录改成用户名 + 密码，应该改哪些模块
3. 如果我要支持复杂路径，例如 `cd a/b/c`，最先应该改哪里
4. 如果我要做断点续传，协议层和文件传输层要怎么变

这时你会发现，第一次只是“看懂”，第二次才是真正开始“理解设计”。

---

## 十一、一句话版建议

如果你只记一条建议，那就是：

**先看公共模块，再看服务端框架，再看会话和命令处理，最后看客户端。**

这个顺序最不容易乱。
